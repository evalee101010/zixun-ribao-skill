# 出海泛娱乐 资讯日报 Skill

把"抓取 → 过滤 → 浓缩 → 排版 → 校验"沉淀成一条可每日重复、可移植（Claude Code / Codex 等都能跑）的流程。

## 适用场景
- 公司主营出海泛娱乐，需每日产出固定格式的资讯简报。
- 覆盖三大板块：泛娱乐·出海、创新·AI娱乐、全球融资。

## 需求基线（给 Codex / 任何 agent 复现，逐条满足）
1. **时间维度**：仅取北京时间（Asia/Shanghai）「前一日16:00 — 当日16:00」的资讯；抬头注明该时间范围。脚本默认按此滚动窗口（`bj_window`），不要跟随电脑显示时区漂移。
1a. **历史日期测试**：如需测试过去某天，运行 `python fetch_news.py --date YYYYMMDD`；窗口仍为北京时间「前一日16:00 — 当日16:00」。列表页源会先进文章页解析发布时间并按窗口过滤；仍无时间戳的 listpage 默认不进入精选/备选正式候选，但会出现在草稿末尾「列表页无时间戳排查」区，避免把当前列表页或旧文误当成当日新闻，同时保留人工核验线索；只有明确排查抓取问题时才加 `--include-undated` 临时放回候选区。
1b. **跨日链接回溯**：抓取时自动回溯前 2 天草稿/成稿链接；若 URL 已在前 2 天出现，本日不再新增。
2. **三大板块 + 赛道**：泛娱乐·出海（休闲游戏/短剧/漫剧/小说·网文/社交·Dating/其他）、创新·AI娱乐（AI模型与应用/创新硬件）、全球融资（泛娱乐融资/泛AI·AI应用融资）。无内容的赛道省略。
2a. **主题准入（宁缺毋滥，治"无关混入"）**：每条先判定其**正文主体**是否属于三类之一——① 出海泛娱乐（游戏/短剧/漫剧/网文/社交·Dating/IP/泛娱乐内容及其业务·财报·数据）；② 娱乐向 AI（面向内容/娱乐/消费者的 AI 模型·应用与创新硬件）；③ 泛娱乐 / 泛AI·AI应用融资。不属于的**直接丢弃**。明确**排除（OUT）**：纯跨境电商/品牌卖货（服装、3C 等带货/选品/GMV）、宏观经济与监管约谈、电诈反诈等社会与政策/案件新闻、半导体/芯片股价行情、航天·火箭·卫星·太空算力、自动驾驶/Robotaxi/Robovan（属具身）、与娱乐无关的企业人事·财经杂讯、新闻早报/资讯聚合体（如《8点1氪》《早报》）。"其他"赛道只收**明确属于泛娱乐**但不在四个子赛道的内容，**不得当作无关内容的兜底桶**。
2b. **归类只看正文、不看来源（治"按渠道分类"）**：板块/赛道由该条正文实际在讲什么决定，与它来自哪个站点/feed 无关；来源仅用于取链接与文末 Source 表。**不得仅因正文出现某关键词（如"社交 App""AI"）就归入对应赛道**（例：一条电诈案件即便提到"社交 App 引流"，也不属于社交·Dating，应整条剔除）。
2c. **同一事件合并（治"同类项不合并"）**：同一主体 + 同一事件被多源报道时，合并为信息最全的一条，多来源链接并列；融资/榜单/财报尤其要合并，不要拆成多条。
2d. **行业大会/活动准入（治"会议预告泛滥"）**：只写"大会召开/报名/议程/嘉宾阵容/注册人数/活动回顾"、但没有实质行业观点、趋势判断、关键数据或公司策略信息的会议类资讯，默认不收。除非是 ChinaJoy 等行业级大会，且原文输出了可复用的行业观点或数据，否则不要占用正文位置；若只是日程信息或会务宣传，整条剔除。
2e. **业务关注提权信号（进入精选/备选复查时优先看）**：以下内容即使脚本 score 不高，也要人工提权复查，避免漏收：
   - **AI 营销 / AI 增长 / 达人带货自动化**：与广告投放、达人增长、营销自动化、带货链路、内容生成到转化效率相关的 AI 应用，属于营销增长类业务关注信号；如 K2Lab / Moras 这类案例，可按使用者业务画像提高优先级。
   - **高流水小游戏 / 小游戏矩阵 / 发行工厂模式**：不是普通单一休闲小游戏，而是具备高流水、矩阵化复制、混休/IAP 转型、发行工厂方法论的公司或产品，应明显高于普通小游戏新闻；如 Voodoo 日流水/矩阵变化。
   - **微信生态下的 AI 支付 / AI 交易基础设施**：微信支付、微信生态与 AI 消费/支付/交易结合的动态需要跟踪，即使不直接属于泛娱乐主题，也可作为交易基础设施动态提高可见度。
   - **头部模型关键进展（短条即可）**：只关注全球范围内最头部的模型或能力方向，尤其视频生成头部模型（如 Seedance）与 LLM/Coding 头部模型（如智谱 GLM 5.2 这类在 Coding 领域进入头部梯队的动态）。非头部、全球范围排不上号的模型不要因"模型发布"本身提权；入选时通常简短提一句即可，不必展开成长条。
   - **OpenAI 等头部 AI 公司关键财务/上市动态**：OpenAI 财报、现金消耗、上市前材料、重大融资等，即使不直接属于泛娱乐或具体应用，也值得保留一眼关注。
3. **单条体例**：简明标题 → 正文（按信息量忠实浓缩，通常 200–400字）→ 原文链接（必带）。融资走模板 C。详见 `format/格式规范.md`。
4. **只浓缩、不做阅读理解**：把原文已有信息（是什么、关键数据、各方说法）提炼出来，**不加入编辑自己的分析/判断/"解读"**；原文里的观点可转述并注明出处。
4a. **去 AI 感（措辞，必做）**：成稿读起来要像人写的新闻条目，不像 AI 复述。两条硬规矩：
   - **不用泛指转述**：禁止 `文章称 / 原文称 / 原文认为 / 文章提到 / 文章将其视为 / 公众号文章称 / 报道回顾称 / 节目摘要称` 这类不指名的转述壳。要么**指名一次**（如 `据36氪`、`白鲸出海称`、`AppMagic 数据显示`），要么**直接陈述事实**；同一条里同一来源不要反复挂"文章""报道"。转述源的观点时也只在句首点一次名。
   - **不补"解读尾巴"/营销腔**：删掉原文没有、由你补的拔高句，典型句式 `这意味着… / 显示… / 反映… / 说明… / 对X而言，这仍是…的重要场景 / 提供了一个…样本 / 强调…的自然观感`。条目止于事实和数据，不替读者下结论；产品描述用中性词，少用 `异军突起、华丽、沉浸、气质介于…之间` 等文学化/广告化形容。
5. **具体产品粒度**：每天纳入若干"单个产品发布/更新"（IT之家/ProductHunt/ai-bot 提供）。
5b. **配图（仅两类）**：榜单/数据图、以产品形态为亮点的产品截图，才配 1 张图；从草稿的"候选图"URL 里挑最具代表性的一张，写在正文后链接前 `![](图片URL)`（源站 URL，飞书粘贴时自动导入）。其余不配图；微信来源图有防盗链不用；脚本没给候选图就不配。注意：codex exec 沙箱可能无网络，只能用草稿里已给的候选图 URL，不要自己去抓图。
5c. **配图正文相关性/图像识别（成稿前必做）**：配图前要做视觉核验，确认图片确实对应正文核心对象。合格图片包括产品内截图、游戏实机画面、真实硬件/产品图、榜单截图、数据图表；不合格图片包括文章封面装饰图、Web 端首页/落地页截图、氛围图、泛用公司宣传图、与正文产品或数据无关的配图。国外媒体常用与正文无关的美化封面，不能因为它是首图就直接采用；无法确认相关性时宁可不配图。
5d. **配图覆盖与可用性自查（成稿前必做）**：成稿后逐条核对——凡榜单/数据类（手游收入榜、短剧投放周榜、收入/流水数据等）与"以产品形态为亮点"的产品条目，草稿给了合格候选图就必须配上一张，漏配的补上并指明对应原文链接。白鲸候选图常以 `www.baijingapp.com/ueditor/...` 给出，该域名会 301 跳转到 `www.baijing.cn/...`，**直接改用 `baijing.cn` 域名写进 `![]()`**，避免飞书导入失败（有网络时可 `curl -I` 抽查 1–2 张返回 200、Content-Type 为 image/*）。非榜单/非产品条目没图属正常，不要硬配。
6. **取全文（关键）**：优先基于**完整正文**浓缩，把原文已写明的主体、背景、数据提炼出来（例：白鲸"120亿短剧独角兽IPO"，正文写明是印度 Kuku Technologies、估值18亿美元、成立2018、创始人来自IIT——这些必须提炼进日报）。白鲸正文在静态 `<p>`、36氪出海在 `__NEXT_DATA__`，脚本 `extract_article` 已自动取（草稿里带 summary）；公众号优先走 WeWe RSS/VBRS 全文，未取到全文时只能保留标题级事实，不得自行扩写。
6a. **公众号"只有标题"时的取全文兜底（治"含糊带过"）**：草稿里某公众号条目只有标题、无 summary，多半不是抓不到，而是该文为**图文混排 + 内嵌 base64 大图**，正文文字被图片/SVG/长 token 噪声淹没导致提取失败（典型：白鲸出海公众号转载，正文 9000+ 中文字却被几十张内嵌图冲散）。补救顺序：① 让 `extract_article`/WeWe 解析时**先剥离 `<svg>`/`<style>`/`data:` 图片块和长 base64/token，再抽正文**；② 仍不行就读**本机 WeWe RSS 全文源** `http://localhost:4000/feeds/all.rss`（`FEED_MODE=fulltext`，按标题定位该 item 的 `description`，同样先去 `data:`/长 token 再留中文）；③ 都不行才退化为标题级保留，并按需求基线 9c 自查是否漏了重要条目。`mp.weixin.qq.com` 直链被前端渲染 + 域名访问限制双重挡住，不要指望直接抓原文。
7. **精选控篇幅**：每赛道取最重要数条，整篇约 ≤30 条；排序优先级 出海/全球 > 有数据或资本动作 > 业务关注提权信号（2e） > 头部主体 > 时效。头部模型关键进展一般短条收录即可，避免挤占更垂直的游戏/短剧/融资条目。
8. **去重**：链接精确 + 跨源标题近重复（脚本已做）+ 前 2 天历史 URL 回溯 + 人工核主题级重复。
8b. **列表页发布时间硬门槛**：白鲸/36氪出海等 `listpage` 必须先进正文页解析发布时间（白鲸 `<time>`、36氪出海 `article:published_time`/`baidu_ts` 等）；能解析的按北京时间窗口过滤并在草稿显示发布时间。日报默认只要当天窗口内内容；人工兜底也只能考虑近 3 天内报道。文章页仍取不到时间的列表页链接，默认不进入精选/备选正式候选，只放进草稿末尾「列表页无时间戳排查」区；只有明确排查时才用 `--include-undated` 调试并标 `⏱无时间戳` 放回候选区。
9. **当前排除方向**：具身/机器人（人形、机械臂、四足、宇树等）与自动驾驶/Robotaxi/Robovan/无人车先不看；"聊天机器人/对话机器人"等 AI 产品照常保留。
9a. **成稿后逐条主题准入复查（治"无关混入"，必做）**：成稿后对每一条再过一遍 2a/2b/9，凡正文主体不属三类的整条删除，删空的赛道连标题一并删。以下为**实际出现过、被误收又必须剔除**的典型（看正文主体，不看是否提到关键词）：电诈/反诈等社会案件（即便提到"社交 App 引流"）、监管约谈/政策解读（如火车票 OTA 约谈、无人驾驶网约车政策误读）、自动驾驶/Robotaxi/Robovan（如曹操出行 RoboX）、半导体/芯片股价行情（如寒武纪创新高）、航天·火箭·卫星·太空算力（如某太空算力天使轮）、与娱乐无关的企业人事/董事会变动（如某 VC 加入 SpaceX 董事会）、新闻早报/资讯聚合体（《8点1氪》《早报》等）。
9b. **归类按"公司当前主体业务"，不按创始人/历史背景（治"误排除"）**："世界模型 / world model / AI 视频·世界生成"属娱乐向 AI（内容生成），即便创始团队出身自动驾驶，也按公司现在做的内容生成业务归入创新·AI娱乐或全球融资，不得因创始人背景误判为"自动驾驶"而排除（例：Odyssey 世界模型融资应纳入）。
9c. **重要信息漏检自查（治"该收没收"）**：成稿后对照草稿"备选"池逐项扫一遍，凡符合三类主题、且属于**重大资本动作（显著金额/估值/IPO，如≥1亿美元或头部机构领投）或核心出海赛道（游戏出海、短剧/漫剧出海等）**的条目，不得仅因脚本 score 低、或只有标题级信息而漏掉；若确不收，须有明确理由（重复/超窗/信息不足）。把"重要信息是否被收录"当作与"无关信息是否被剔除"同等的校验标准。
10. **Source 来源收尾**：正文结束后用 `Source 来源` 表统计本期实际采用的来源与条数；不再放“个人社媒”“兴趣产品跟踪”和 appendix。
11. **正文不写采集信息或内部过程话术**：单条正文只写资讯事实，不写“该条来自/经 WeWe RSS/VVRS/VVRSS/VBRS 抓取/来源为 XXX”“补足观察信号”“以原文为准”“待补全文”“草稿摘要显示”“公开摘要未展开”“该条作为……保留”“后续可继续跟踪”等采集来源、兜底说明、筛选理由或后续动作；来源只进入文末 `Source 来源` 表。
12. **格式硬约束**：URL 保持裸链不转义；不同一级/二级标题之间用独立一行 `<br>` 做真实空行；标题图标、板块顺序、Source 表格式以 `format/格式规范.md` 为准。
13. **输出文件**：同一天只更新同一个 `资讯类简报-{月.日}日.md`，不要生成多个成稿文档。
14. **自动化原则**：定时任务只负责触发本 skill；完整规则必须写在本 skill 与 `format/格式规范.md` 中，不把业务规则复制进本地自动化 prompt。

## 源分层策略（资源效率最优）
不要试图每日自动爬全部上百个渠道——多数 ROI 很低。按三层用：
- **Tier 1 主力聚合/全文源**：已替你浓缩好或带完整正文的 feed，一个顶一片，自动按标题分流到三大板块：
  - **ai-bot.cn/每日AI资讯**：纯静态，每条"标题+100~200字+来源链接"，覆盖 AI 模型/应用/AI娱乐/融资。
  - **IT之家 RSS**（`ithome.com/rss/`）：60条/带完整正文/带时间戳，**产品粒度细（单个产品发布/更新）**，补"具体产品"维度。
  - **36氪主站 feed**（`36kr.com/feed`）：带数千字完整正文的**深度文**，便于忠实浓缩出信息完整的条目。
  - **Product Hunt**（`producthunt.com/feed`）：每条=一个新产品，最适合"具体产品"维度（英文、偏 SaaS，靠关键词降噪）。
- **Tier 2 垂直补充源**：补 Tier1 覆盖不到的"出海泛娱乐业务/数据"面——游戏出海（PocketGamer/触乐/GameLook）、短剧（DataEye）、出海综合（白鲸出海/36氪出海 listpage）、数据站（Sensor Tower/点点）。RSS/列表页自动 + 公众号人工。
- **Tier 3 选题池**：附件《信息渠道汇总.md》里的即刻/X/领英/播客/Newsletter 等大列表，噪声大且多需登录，**仅作人工选题与深挖灵感，不做每日自动抓**。
- **语鲸 lingowhale**：前端渲染、抓取拿不到内容，定位为人工订阅公众号合集的工具，不纳入自动管道。

| 渠道 | 稳定通道 | 状态 |
|---|---|---|
| **白鲸出海** | 快讯页 `/newsflashes` 列表解析；**正文藏在 `/article` 静态 `<p>` 里**（"请开启JS"只是提示壳），脚本自动取全文（公司名/背景/数据都在正文）并从文章页 `<time>` 解析发布时间。⚠️ 别用 `/feed`（陈旧问答子站）。还转载 DataEye 短剧周榜 | ✅ listpage + 自动取正文 + 发布时间 |
| **36氪出海** letschuhai | 列表解析；**正文在 Next.js `__NEXT_DATA__` 的 `content.rendered`**，脚本自动取全文，并从 `article:published_time` / `baidu_ts` 等 meta 字段解析发布时间 | ✅ listpage + 自动取正文 + 发布时间 |
| **36氪主站**（快讯/融资） | SPA，公开 RSSHub 被封(403)；需自建 RSSHub 用 `/36kr` 路由 | ⚠️ 待自建 |
| **DataEye** | 官网 SPA、短剧数据在 JS/登录后，简单抓取拿不到；周榜常被白鲸转载可间接获取，或 Chrome 渲染/找 API | ⚠️ 公众号/间接 |
| **短剧自习室** | 纯公众号、无官方 API | ⚠️ 人工/WeWe RSS |

结论：**白鲸出海 + 36氪出海** 两个 listpage 源已能稳定自动覆盖出海泛娱乐七七八八（实测合计 60+ 条/天，含游戏买量、海外短剧榜单、出海数据），且会进文章页补发布时间用于窗口过滤；DataEye/短剧自习室作公众号人工补充；36氪主站待自建 RSSHub。

## 每日执行流程（agent 按此走）

1. **抓取**：运行 `python fetch_news.py`，生成 `资讯日报-草稿-YYYYMMDD.md`（候选清单 + 公众号待补清单 + 告警）。不要用 `--hours 24` 跑正式日报；`--hours` 仅用于临时测试，否则会偏离北京时间 16:00—16:00 窗口。历史日期测试用 `python fetch_news.py --date YYYYMMDD`。
2. **补公众号**：草稿里"⚠️ 公众号待人工贴链接"列出的源，让用户把当日链接/正文贴进来（公众号无官方 API，这步暂时人工）。其它公开网页若需补充可用 web 检索/抓取。
3. **筛选**：从候选里挑出符合三大板块、有信息量的条目；去重、去边角料。出海/全球/融资优先。草稿里的 ⭐精选 是默认采用清单，成稿时必须逐条覆盖校验；若某条不采用，只能因为跨源/跨日重复、超出时间窗、与主题明显无关或信息质量不足，并在工作中明确剔除原因，不能因为摘要短、公众号正文未展开或赛道较短而直接漏掉。若只拿到标题和链接，保留标题级事实并避免扩写。
4. **取全文**：白鲸/36氪出海的正文由脚本 `extract_article` 自动抓取（草稿里已带 summary，含主体/背景/数据）；公众号优先用 WeWe RSS/VBRS 输出全文，必要时再人工/Chrome 补全。务必基于正文写；若只有标题和链接，正文只写标题能支撑的事实，不写采集说明或兜底话。
5. **浓缩排版**：逐条按 `format/格式规范.md` 套模板，**忠实浓缩原文、不加自己的解读**——
   - **模板 A 分维度标签型**：产品/财报类、信息维度多的业务资讯。
   - **模板 B 概述段落型**：单点消息。
   - **模板 C 融资型**：两段叙述，全部融资条目用，归入「全球融资」板块。
   - 每条：简明标题 → 正文（按信息量浓缩，通常 200–400字）→ 原文链接（必带）。
6. **组装**：复制 `format/日报模板.md` 骨架，填入各条，删掉当日无内容的赛道，结尾补 `Source 来源` 表（只统计最终成稿正文实际采用的来源与条数）。
7. **生成后质检**：成稿后运行 `python check_report.py --draft 资讯日报-草稿-YYYYMMDD.md --report ../资讯类简报-{月.日}日.md --write 资讯日报-质检-YYYYMMDD.md`。质检报告留在 skill 目录，不进入最终日报正文。
8. **修复再确认**：若质检出现 `FAIL`，必须先修正成稿再重新运行质检；`WARN` 需要人工回看草稿摘要/原文链接确认。只有质检无 `FAIL` 才算完成。
9. **产出**：保存/更新当天唯一的 `资讯类简报-{月.日}日.md`，不要另起多个成稿文件。

## 校验清单（产出前必过）
- [ ] 抬头注明时间范围：北京时间 前一日16:00 — 当日16:00；超窗条目已剔除；列表页无时间戳条目未默认混入精选/备选正式候选，若出现只应在排查区。
- [ ] 正式抓取使用 `python fetch_news.py` 的默认北京时间窗口；未用 `--hours 24` 代替。历史测试可用 `--date YYYYMMDD`，并确认正文页仍无时间戳的列表页条目只进入排查区、未混入当前候选；人工兜底也不收超过近 3 天的旧报道。
- [ ] 已精选排序：每赛道取最重要数条，整篇约 ≤ 30 条，按重要性（出海>数据/资本>头部主体>时效）排序。
- [ ] ⭐精选覆盖校验：草稿中每条 ⭐精选 都已进入成稿，或已因重复/超窗/明显无关/信息不足被明确剔除；WeWe RSS/公众号条目即使只有标题+链接，也不能无声跳过，只能保留标题级事实并避免扩写。
- [ ] 业务关注提权复查（按 2e）：已扫过 AI 营销/增长、小游戏工厂/高流水矩阵、微信生态 AI 支付、头部视频模型/LLM Coding 模型、OpenAI 关键财务/上市动态；符合条件者未因脚本 score 低而漏收。头部模型只收全球头部，非头部模型发布不提权；入选时通常短条即可。
- [ ] 已去重：同一事件多源只留信息最全一条（脚本两级去重 + 前 2 天 URL 回溯 + 列表页无时间戳默认不进正式候选 + 人工核主题级重复）。
- [ ] 篇幅合理：按信息量忠实浓缩（通常 200–400 字），不注水；前端渲染源（白鲸/36氪出海/公众号）已取全文再写，不是只凭标题。

- [ ] 主题准入：每条正文主体确属三类之一；已剔除跨境电商卖货、宏观/监管约谈、电诈等社会案件、半导体股价、航天火箭/太空算力、自动驾驶/Robotaxi、与娱乐无关的企业人事财经杂讯、新闻早报/资讯聚合体等无关内容（成稿后已按 9a 逐条复查，删空赛道连标题删）。

## 升级路径
- **第一阶段（当前）**：半自动。公开 RSSHub 实例 + 各站 RSS 自动抓，公众号人工贴。
- **公众号自动化（WeWe RSS）**：`wewe-rss/` 目录已备好 `docker-compose.yml` 与部署指引。`docker compose up -d` → http://localhost:4000 扫码登录微信读书 → 订阅公众号 → 把 `sources.yaml` 里「微信公众号(WeWe RSS)」源的 `url` 填成 `http://localhost:4000/feeds/all.rss`、`enabled: true`，一个 all 源收下全部公众号、按标题自动分流，替代 `type: wechat` 的人工源。注意微信读书约 10 公众号/账号、有索引延迟、可能被封控，保留人工兜底。
- **第二阶段（RSSHub）**：自建 RSSHub（Docker/VPS），把 `sources.yaml` 的 `rsshub_base` 改成自建地址，提升 36氪主站等 RSSHub 路由的稳定性。
- **融资增强**：点点数据 API 上线后，在 `.env` 填 `DIANDIAN_API`，把对应源 `enabled: true`，扩展脚本 api 分支。

## 注意
- 公众号抓取无官方 API，任何第三方方案都可能失效，始终保留"人工贴链接"兜底。
- 抓取仅用于内容聚合，遵守各站 robots 与版权，简报中保留原文出处链接。
